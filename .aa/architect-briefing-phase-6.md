# Phase 6 — Phase 0 Generator — Architect Briefing

## Project
Go coding-agent harness. Full spec: `agent-harness_spec.md`. Before starting, read all five prior ADRs and `phase-N.log` files. This phase builds the meta-agent described in spec §13: given a project's `*_spec.md`, it decides per-phase stage selection and writes kickoff prompts + loop-iteration templates for the agents that will actually build that project.

## The reference examples already exist
This harness's own five completed phases were bootstrapped by hand, following exactly the pattern this phase needs to automate: read the spec → decide which agents a phase needs → write a kickoff briefing per agent per phase, each referencing prior outputs by path (`docs/adr-phase-N-<name>.md`, `phase-N.log`) rather than embedding content, per the Inter-Agent Handoff Convention (spec §12.1). The `docs/adr-phase-*.md` and `phase-*.log` files already in this repo are real, working examples of what downstream agents successfully consumed — study their structure as the target output shape, not just the spec's abstract description of it.

## Your task this phase

1. **Input handling.** Design how the Phase 0 agent receives a project's spec file — a path argument, most likely (`--spec path/to/project_spec.md`), read via the existing `read_file` tool. Specify how this differs from the harness's normal per-agent invocation (this agent doesn't need `edit_file`/`bash_exec` on arbitrary project code — it needs `read_file`, `list_dir`, and `create_file`/`edit_file` scoped only to wherever templates get written).

2. **Stage-selection output format.** Design the manifest that records "which of the five agents does phase N need" — per spec, not every phase needs every agent (§13.1's own table is the reference example: Phase 5 needed only builder+tester, Phase 3 needed architect+librarian+builder+tester). Specify the concrete file format (a YAML or JSON manifest under `.aa/templates/manifest.yaml` is a reasonable default — justify your choice) and its exact shape (phase number → agent list, at minimum).

3. **Kickoff prompt generation.** Design how the agent produces one kickoff-briefing file per phase per selected agent, following the naming convention already established (`architect-briefing-phase-N.md`, etc.) and the structural pattern visible in this repo's own `.aa/` briefings: project context + read-first instructions pointing at specific prior-phase paths + task breakdown + explicit "out of scope" section + deliverable section specifying exact output paths. This is the part with real design/prompt-engineering surface — specify enough about the *system prompt* this meta-agent runs under (not user prompt — that's Phase 0 invocation-specific) that builder can get a system prompt file (`agents/prompt-generator.md`, matching the existing agent config convention from Phase 4) that reliably produces briefings with this shape, not free-form prose.

4. **Loop-iteration template generation.** Per spec §12, most agents besides tester get a generic `<agent>-template.md` for loop iterations (filled via `text/template` at runtime from `events`/`files` data, no LLM call). Specify what this agent needs to produce here — likely much simpler than kickoff prompts, since these are mostly fixed structurally across phases (the placeholder slots are what varies, and even those are fairly standard: "last test output," "prior diff," "files touched"). Confirm whether Phase 0 needs to generate these fresh per project, or whether a small set of genuinely generic templates (independent of project specifics) can just live as embedded defaults from Phase 4's bootstrap system rather than being regenerated by Phase 0 every time — this could meaningfully simplify this phase's scope if the loop-templates turn out not to need per-project customization at all.

5. **Per-phase agent-set validation.** Since stage selection can omit agents, specify how the harness (in later real usage, not this phase's own tests) would know which config file to load for a given phase — presumably it just looks up the manifest from item 2 before invoking any agent. Confirm this phase's deliverable includes writing that manifest correctly, even though *consuming* it to actually drive phase execution is arguably a `cmd/harness` integration concern you should scope explicitly in or out.

## Explicitly out of scope for Phase 6
- Spec audit implementation (§14) — Phase 7
- Actually wiring `cmd/harness` to consume the stage-selection manifest and auto-run phases end-to-end — decide explicitly whether this is in scope per item 5, but lean toward out-of-scope unless it's a small addition, since the spec's own rollout plan treats this project's phases as hand-dispatched regardless
- Any change to `internal/llm`, `internal/store`, `internal/tools`, `internal/loop`, `internal/config`, `internal/gitops`

## Deliverable
Write your design to `docs/adr-phase-6-prompt-generator.md`. Cover all five items, plus a concrete worked example: show what the Phase 0 agent's output would look like if given a short, made-up test spec — this makes the design testable and unambiguous for tester later. Append a `phase-6.log` entry.
